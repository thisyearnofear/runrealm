# H3 Territory Migration Plan

Status: design only. Deployed contracts in
`contracts/crosschain/RunRealmUniversal.sol` and
`contracts/libraries/GameLogic.sol` are NOT being changed in this step.
The on-chain identifier remains `geohash` (a 6-12 char string) until a
deliberate upgrade is shipped.

## Why H3

The current `geohash` field is a coarse lat/lng string (`{lat}_{lng}`
with 6 decimal places) generated by `TerritoryService.generateGeohash`.
This produces large, overlapping bounding boxes that:

1. Don't match the actual area a runner covered — a 400m straight line
   through a 50m-wide geohash cell is recorded as 50m of "coverage".
2. Have jagged, axis-aligned boundaries that produce visual artefacts
   when rendered on the map and don't match H3-native players' mental
   model of territory shape.
3. Can't represent a single run as multiple discrete claimable cells
   without manual polygon math on the client.

H3 (Uber's hex grid) gives uniform hexagons at a chosen resolution with
clean neighbour relationships, set-equality (two territories are "the
same cell" iff their H3 indices match), and `gridDisk` for "neighbours
within N rings" queries the game UI already wants.

## Resolution choice

`H3_RESOLUTION = 9` — ~0.105 km² average hex area, ~174m edge length.
This lives in `packages/shared-core/utils/h3-territory.ts` as the
single source of truth and is re-exported through `h3Summary()`.

## Migration phases

### Phase 0 (shipped) — client-side, additive

- New `TerritoryCell` type, `coordsToCell`, `routeToCells`,
  `cellToPolygon`, `cellsIntersect`, `neighboringCells` in
  `packages/shared-core/utils/h3-territory.ts`.
- `Territory` extended with optional `h3Cells?: TerritoryCell[]` and
  `h3Resolution?: number`. Existing reads of `geohash` still work.
- `TerritoryService.attachH3Cells(territory, run)` is called from
  `createTerritoryFromRun` so every newly-created territory is populated
  with H3 metadata without any contract change.
- `TerritoryService.h3Summary(territory)` is the read-only shape exposed
  to the server (in `/api/runs`) and the map layer.

### Phase 1 — server-side validation

The `POST /api/runs` stub added in step 2 should:

- Accept an optional `h3Cells: string[]` (the H3 indices) in the request
  body alongside the existing `geohash` field.
- Validate each H3 index via `h3-js.isValidCell` and reject the request
  with 400 if any are malformed.
- Persist the H3 indices alongside the run row in whatever store the
  server grows into (the current stub has no DB; this becomes work when
  a DB is added).

### Phase 2 — contract upgrade (future, not in this step)

The deployed `RunRealmUniversal` and `GameLogic` contracts key
territories on a string geohash. Migrating to H3 on chain requires:

1. Add `mapping(uint64 => uint256) private _h3ToTokenId;` to
   `RunRealmUniversal` (H3 indices fit in uint64).
2. Keep `_geohashToTokenId` for backward-compat reads of pre-upgrade
   territories.
3. In `GameLogic.validateTerritory`, accept an `enum TerritoryIdType
   { GEOHASH, H3 }` parameter and dispatch to the right mapping.
4. `IRunRealmGame` gets a new `claimTerritoryByH3(uint64 h3Index, ...)`
   entry point. The existing `claimTerritory(string geohash, ...)` stays
   for old clients.
5. A new event `TerritoryCreatedByH3(uint256 indexed tokenId, uint64
   indexed h3Index, ...)` is emitted.

Gas: `mapping(uint64 => uint256)` is meaningfully cheaper than the
existing `mapping(string => uint256)` (no string storage, no hashing in
the SLOAD path). For territory-heavy seasons this is a measurable win.

### Phase 3 — deprecation

After one full season on H3, `geohash` becomes the legacy field:

- Server stops accepting it on writes.
- Contract `claimTerritory(string geohash, ...)` is annotated
  `@deprecated` and gated behind a feature flag.
- Old territories still resolve via the dual mapping for read-only
  history views; the map render path filters them out after a season
  cutoff.

## What this step does NOT do

- Does not modify any contract.
- Does not change the on-chain identifier.
- Does not require the server to validate H3 yet (the stub accepts the
  field but doesn't require it).
- Does not add the H3 cells to the map render — that's step 5
  (React shell + map canvas).
